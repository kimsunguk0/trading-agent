# 운영 매뉴얼 (§21.4 기준)

## 1) 장 시작 전 체크리스트 (08:30 KST)

1. `docker compose ps` 및 핵심 헬스체크
   - api, worker-execution, worker-strategy, worker-monitor, news/collector, reconciliation
2. DB/Redis 스키마 존재 확인
   - `trading_paper` 또는 `trading_live` 스키마, `trading_*` 이벤트 스트림 접속
3. 가격/체결 데이터 신선도 확인
   - 30분 이내 OHLCV/티커 스냅샷 존재
4. `SystemState` 상태 확인 (NORMAL 또는 허용된 degraded)
5. 리스크 한도/일간 손실 상태 확인
   - 일간 손실 한도 잔여율 80% 이하인지 점검
6. 브로커 계정·토큰 상태 확인
   - 브로커 크리덴셜/만료 시간 점검
7. ChatOps 접근 채널 상태
   - Telegram bot heartbeat, `/status` 응답 확인

## 2) 장 마감 후 체크리스트 (15:40 KST)

1. 장 마감 이벤트 반영 완료 확인
2. 마감 후 주문/잔고 재동기화(Reconciliation) 1회 수행
3. 금일 체결 건수/슬리피지/취소율 집계 후 이상치 점검
4. 전략별 승인 상태/대기 오더 정리
5. `daily_briefing`이 전송되었는지 확인
6. 당일 위험 노출률 상향 항목(섹터, 전략별, 마켓별) 점검
7. 로그/알림 보존 및 장애 티켓 라우팅

## 3) EMERGENCY_STOP 복구 절차 (7단계)

1. `SystemState` 전환 원인 식별 (`EMERGENCY_STOP` 이벤트 원천 확인)
2. 위험 원인별 격리: 거래 전송, 승인 채널, 전략 엔진, 시장 데이터
3. 신규 주문 차단 확인(오더 게이트 정지)
4. 오더 미체결/미확인 주문을 reconciliation으로 추적하여 상태 정합성 확보
5. 공통 상태를 `HALTED` 또는 `DEGRADED_*`로 완화 후 운영 영향 범위 제한
6. 원인 조치(리스크 한도 수정, LLM/브로커/API 복구, 캐시 초기화, 토큰 갱신)
7. 운영 책임자 승인 후 `EMERGENCY_STOP` 해제 및 `system_state` 전환 로그 보관

## 4) 잔고 불일치 발생 시 절차

1. 거래소/브로커 잔고와 `portfolio` 테이블 잔고를 즉시 비교
2. 오차 금액과 심볼별 누락/과대 집계 분해
3. 미확인 오더(`UNKNOWN_SUBMITTED`)를 reconciliation 대상으로 수집
4. 15분 이내 교차검증이 되지 않으면 수동 모드로 전환
5. 감사지표 기록(오차 시퀀스, 주문 ID, 전략 ID, 브로커 응답)
6. 재동기화 배치 실행 후 샘플 10종목 재확인
7. 회복 후 장 외 시간에 포스트모템 로그 아카이브

## 5) LLM 서버 다운 시 절차

1. `allow_llm_calls` 상태 전환 및 DEGRADED_LLM 플래그 설정
2. RULE 기반 경로(미지원 전략 제외)만 유지하도록 필터링
3. 이벤트 큐는 유지하되 승인 대기 이벤트 생성 중단
4. 백업 응답 모드로 전환(최근 캐시/룰 캐시 우선)
5. LLM 서버 정상화 후 승인/캐시 유효성 검증
6. 회복 후 전략별 재활성화 순서는 PAPER → LIVE_APPROVAL 전략부터

## 6) 브로커 API 장애 시 절차

1. 브로커 헬스체크 실패 감지 시 `BROWNOUT` 또는 `HALTED` 전환
2. 주문/체결 조회 호출 중단 및 신규 진입 제한
3. 오더 상태 불명 건은 `reconciliation` 큐로만 추적
4. 대기 주문/미체결 주문 수동 조회를 제한 모드에서 제한적으로 수행
5. 브로커 장애 알림을 운영 채널에 공지 후 원인(자격증명/네트워크/쿼터)
6. API 정상화 후 샘플 주문 1건 승인/상태조회 연동 테스트
7. 수동 승인 대상/자동 전략 활성도 복구

## 7) 일일 손실 한도 도달 시 절차

1. 리스크 관리자에서 `daily_loss_limit_reached` 이벤트 확인
2. 신규 주문 즉시 중단, 기존 보유 포지션은 정책에 따라 단계별 이익 실현 검토
3. 전략별 가중치/노출 상한을 일시적으로 0 또는 보수값으로 조정
4. 손실 원인 분석(슬리피지, 시장 충격, 모델 오탐지) 태깅
5. 일일 운영 일지에 운영자의 승인/해제 조건 기록
6. 다음 세션까지 `HALTED` 또는 `DEGRADED_MARKET` 상태 유지
